Every request to the PayPac API is authenticated and authorized against a single role stored on the `User` record. Roles are enforced at the route level by the `authorizeRoles` middleware, which compares the authenticated user's role against the list of roles permitted for that endpoint.

## The six roles

| Role | Description |
|------|-------------|
| `PAYPAC` | Platform super-admin. Full access to every endpoint, including status transitions, administrative overrides, and all transaction data. |
| `ORGANIZER` | Event organizer. Creates and manages their own events, assigns staff, and views invoices and ticket stats for events they own. |
| `STAFF` | Event staff assigned to a specific event. Can validate tickets at the door after completing event check-in. |
| `STAFF_PROMOTER` | A staff member who also has promoter capabilities. Shares validation rights with `STAFF` and ticket transfer rights with `PROMOTER`. |
| `PROMOTER` | External ticket promoter. Can browse events, manage their own tickets, and earn commission tracked via `PromoterCode`. |
| `CUSTOMER` | End-user buyer. Can purchase tickets, view their wallet, transfer tickets, and register a device public key. |

Roles are stored as the `ROLES` enum on the `User.role` field and default to `CUSTOMER` at registration.

## Role assignment

The role is set directly on the `User` model:

```prisma
model User {
  role  ROLES  @default(CUSTOMER)
  // ...
}

enum ROLES {
  PAYPAC
  ORGANIZER
  STAFF
  STAFF_PROMOTER
  PROMOTER
  CUSTOMER
}
```

A user has exactly one role at any time. Role changes require a direct update to the `User` record, typically performed by a `PAYPAC` administrator.

## Section permissions

Beyond route-level authorization, the `RoleSectionPermission` model provides fine-grained UI section access control. Each role can have a permission record per `Section` with five independent flags:

```prisma
model RoleSectionPermission {
  role        ROLES
  section_id  Int
  can_view    Boolean @default(false)
  can_create  Boolean @default(false)
  can_edit    Boolean @default(false)
  can_delete  Boolean @default(false)
  can_export  Boolean @default(false)

  @@id([role, section_id])
}
```

| Flag | Controls |
|------|----------|
| `can_view` | Whether the role can see the section |
| `can_create` | Whether the role can create records in this section |
| `can_edit` | Whether the role can modify existing records |
| `can_delete` | Whether the role can delete records |
| `can_export` | Whether the role can export data from this section |

Sections are hierarchical — a `Section` can have a `parent_id` pointing to another `Section`, allowing nested menu structures with inherited permission checks.

## DocType enum

When users provide identity documents (for ticket purchases and invoices), the `type_doc` field uses the `DocType` enum:

| Value | Document type |
|-------|---------------|
| `CC` | Cédula de Ciudadanía — Colombian national ID |
| `CE` | Cédula de Extranjería — Colombian foreign resident ID |
| `PA` | Pasaporte — passport |
| `TI` | Tarjeta de Identidad — Colombian minor's ID |
| `NIT` | Número de Identificación Tributaria — business tax ID |
| `SSN` | Social Security Number — US national ID |

The `DocType` appears on both the `User` model (`type_doc`) and on the `Invoice` model (`user_type_doc`), where it is snapshotted at the time of purchase.
