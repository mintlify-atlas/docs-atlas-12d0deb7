---
title: "Tickets"
description: "Manage, transfer, and validate event tickets"
---

Tickets are issued to customers after a successful invoice payment. Each ticket carries a cryptographic token used for QR-based or NFC-based entry validation. Tickets can be transferred, sold, or gifted to other users and maintain a full audit trail via [Ticket Transactions](/api/ticket-transactions).

## TicketStatus enum

| Value | Description |
|---|---|
| `PENDING` | Awaiting payment |
| `PAID` | Payment confirmed, ticket ready |
| `ACTIVE` | Active and ready to use |
| `USED` | Scanned and admitted at the event |
| `FROZEN` | Locked pending a transfer or sale |
| `TRANSFERRED` | Ownership transferred to another user |
| `CANCELED` | Canceled by owner or PAYPAC |
| `EXPIRED` | Past event date, no longer valid |
| `ON_SALE` | Listed for peer-to-peer sale |

## GET /api/tickets/admin

**Roles:** `PAYPAC`, `ORGANIZER`

Paginated list of all tickets. `PAYPAC` sees every ticket across all events; `ORGANIZER` sees only tickets belonging to their own events.

### Query parameters

<ParamField query="event_id" type="number">
  Filter tickets by event ID.
</ParamField>

<ParamField query="status" type="string">
  Filter by ticket status. One of `PENDING`, `PAID`, `ACTIVE`, `USED`, `FROZEN`, `TRANSFERRED`, `CANCELED`, `EXPIRED`, `ON_SALE`.
</ParamField>

<ParamField query="search" type="string">
  Free-text search across ticket reference, booking number, or customer name.
</ParamField>

<ParamField query="from" type="string">
  ISO 8601 datetime. Return tickets created on or after this date.
</ParamField>

<ParamField query="to" type="string">
  ISO 8601 datetime. Return tickets created on or before this date.
</ParamField>

<ParamField query="page" type="number" default="1">
  Page number for pagination.
</ParamField>

<ParamField query="limit" type="number" default="50">
  Results per page.
</ParamField>

### Response

<ResponseField name="data" type="object[]">
  Array of ticket objects.
  <Expandable title="ticket fields">
    <ResponseField name="id" type="number">Ticket ID.</ResponseField>
    <ResponseField name="event_id" type="number">Associated event ID.</ResponseField>
    <ResponseField name="customer_id" type="number">ID of the current ticket owner.</ResponseField>
    <ResponseField name="reference_ticket" type="string">Unique alphanumeric reference.</ResponseField>
    <ResponseField name="booking_ticket" type="string">Booking number (PYC-prefixed).</ResponseField>
    <ResponseField name="status_ticket" type="string">Current ticket status.</ResponseField>
    <ResponseField name="ev_name" type="string">Snapshot of the event name at purchase time.</ResponseField>
    <ResponseField name="ev_date_event" type="string">Snapshot of the event date.</ResponseField>
    <ResponseField name="loc_name_locality" type="string">Locality name.</ResponseField>
    <ResponseField name="ticket_first_time" type="number">`1` = not yet used, `0` = already used.</ResponseField>
    <ResponseField name="created_at" type="string">ISO 8601 creation timestamp.</ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="total" type="number">Total number of matching tickets.</ResponseField>
<ResponseField name="page" type="number">Current page.</ResponseField>
<ResponseField name="limit" type="number">Results per page.</ResponseField>

### Errors

| Code | Reason |
|---|---|
| `401` | Missing or invalid authentication token |
| `403` | Caller does not have `PAYPAC` or `ORGANIZER` role |
| `500` | Internal server error |

<CodeGroup>

```bash cURL
curl --request GET \
  --url "https://api.paypac.com.co/api/tickets/admin?event_id=42&status=PAID&page=1&limit=25" \
  --header "Authorization: Bearer <token>"
```

</CodeGroup>

## PATCH /api/tickets/admin/:id/status

**Role:** `PAYPAC`

Override the status of any ticket. Changing status to `CANCELED` is irreversible through this endpoint.

### Path parameters

<ParamField path="id" type="number" required>
  Ticket ID.
</ParamField>

### Body

<ParamField body="status" type="string" required>
  New ticket status. One of `PENDING`, `PAID`, `ACTIVE`, `USED`, `FROZEN`, `TRANSFERRED`, `CANCELED`, `EXPIRED`, `ON_SALE`.
</ParamField>

### Response

<ResponseField name="message" type="string">Confirmation message including the new status value.</ResponseField>
<ResponseField name="ticket" type="object">
  Updated ticket object.
  <Expandable title="ticket fields">
    <ResponseField name="id" type="number">Ticket ID.</ResponseField>
    <ResponseField name="status_ticket" type="string">New status value.</ResponseField>
    <ResponseField name="updated_at" type="string">Update timestamp.</ResponseField>
  </Expandable>
</ResponseField>

### Errors

| Code | Reason |
|---|---|
| `400` | Invalid status value |
| `401` | Missing or invalid authentication token |
| `403` | Caller does not have `PAYPAC` role |
| `404` | Ticket not found |

<CodeGroup>

```bash cURL

  --url "https://api.paypac.com.co/api/tickets/admin/123/status" \
  --header "Authorization: Bearer <token>" \
```

## PATCH /api/tickets/:id/public-key

**Roles:** `CUSTOMER`, `PAYPAC`

Registers the attendee's device RSA public key on a ticket to enable NFC-based entry validation. Only the current ticket owner can register a key.

|---|---|

| `401` | Missing or invalid authentication token |

<CodeGroup>

```bash cURL

  --url "https://api.paypac.com.co/api/tickets/123/public-key" \
```

## GET /api/tickets/:id/totp-secret

## POST /api/tickets/:id/transfer

**Roles:** All authenticated roles

Initiates a ticket transfer, peer-to-peer sale, or gift to another user. The ticket token is regenerated, device credentials from the previous owner are cleared, and a [TicketTransaction](/api/ticket-transactions) audit record is created. Sales begin with `status_ticket = PENDING` in the TicketTransaction; transfers and gifts complete immediately.

### Path parameters

<ParamField path="id" type="number" required>
  Ticket ID to transfer.
</ParamField>

### Body

<ParamField body="to_user_id" type="number" required>
  Internal user ID of the recipient.
</ParamField>

<ParamField body="to_user_uid" type="string" required>
  Firebase UID of the recipient.
</ParamField>

<ParamField body="to_user_id_phone" type="string" required>
  Phone UUID of the recipient's device.
</ParamField>

<ParamField body="from_user_id_phone" type="string" required>
  Phone UUID of the sender's device.
</ParamField>

<ParamField body="transaction_type" type="string" required>
  Transfer mode. One of `transfer`, `sale`, or `gift`.
</ParamField>

<ParamField body="description" type="string">
  Optional message to attach to the transfer.
</ParamField>

### Response

<ResponseField name="success" type="boolean">Always `true` on success.</ResponseField>
<ResponseField name="message" type="string">Human-readable confirmation.</ResponseField>
<ResponseField name="ticket" type="object">Updated ticket with the new owner's details and a regenerated token.</ResponseField>

### Errors

| Code | Reason |
|---|---|
| `400` | Ticket not found, not owned by caller, already used, or in a non-transferable status |
| `401` | Missing or invalid authentication token |

<CodeGroup>

```bash cURL
curl --request POST \
  --url "https://api.paypac.com.co/api/tickets/123/transfer" \
  --header "Authorization: Bearer <token>" \
  --header "Content-Type: application/json" \
  --data '{
    "to_user_id": 456,
    "to_user_uid": "firebase-uid-456",
    "to_user_id_phone": "device-uuid-456",
    "from_user_id_phone": "device-uuid-123",

```

## POST /api/tickets/:id/validate

**Roles:** `PAYPAC`, `ORGANIZER`, `STAFF`, `STAFF_PROMOTER`

Validates a ticket at event entry. Three validation methods are supported:

1. **QR token** — standard `token_ticket` value from the QR code.
2. **QR token + device UUID** — adds device-binding check to detect screenshot fraud.
3. **TOTP** — 6-digit rolling code from the attendee's app (pass via a separate NFC endpoint).

`STAFF` and `STAFF_PROMOTER` must have checked in to the event. `ORGANIZER` must own the event. Validation only succeeds within the event's check-in window (`date_checkin_open` – `date_checkin_close`).

### Path parameters

<ParamField path="id" type="number" required>
  Ticket ID (used for routing; the actual lookup resolves the ticket from `qr_token`).
</ParamField>

### Body

<ParamField body="qr_token" type="string" required>
  Ticket token string from the QR code. Minimum 32 characters.
</ParamField>

<ParamField body="event_id" type="number" required>
  ID of the event where validation is taking place.
</ParamField>

### Response

<ResponseField name="success" type="boolean">Always `true` on success.</ResponseField>
<ResponseField name="message" type="string">Welcome message for the attendee.</ResponseField>
<ResponseField name="ticket" type="object">Updated ticket with `ticket_first_time = 0` and `used_at` populated.</ResponseField>
<ResponseField name="scanner_id" type="number">User ID of the scanner who validated the ticket.</ResponseField>
<ResponseField name="scanner_role" type="string">Role of the scanner.</ResponseField>

### Errors

| Code | Reason |
|---|---|
| `400` | Token invalid, ticket already used, wrong event, check-in window closed, or scanner not assigned to event |
| `401` | Missing or invalid authentication token |
| `403` | Caller role cannot validate tickets |

<CodeGroup>

```bash cURL
curl --request POST \
  --url "https://api.paypac.com.co/api/tickets/123/validate" \
  --header "Authorization: Bearer <token>" \
  --header "Content-Type: application/json" \
  --data '{
    "qr_token": "a3f9b2c1d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3",
    "event_id": 42
  }'
```

</CodeGroup>
