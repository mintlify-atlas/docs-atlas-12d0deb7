## How authentication works

PayPac uses **Firebase Authentication** for identity management. The backend never handles passwords or login sessions directly. Instead:

1. Your client app signs the user in via the Firebase SDK.
2. Firebase returns an **ID token** (a signed JWT).
3. Your app sends that token to the PayPac API in the `Authorization` header.
4. The backend verifies the token using **Firebase Admin SDK** (`verifyIdToken`), looks up the user in the database by their `firebase_uid`, and attaches the user record to the request.

<Note>
  Login is handled entirely on the client side using Firebase. The PayPac API has no `/auth/login` endpoint — it only validates tokens that Firebase has already issued.
</Note>

## Getting a Firebase ID token

Sign the user in with the Firebase client SDK, then call `getIdToken()`:

<CodeGroup>

```javascript JavaScript
import { getAuth, signInWithEmailAndPassword } from "firebase/auth";

const auth = getAuth();
const userCredential = await signInWithEmailAndPassword(
  auth,
  "user@example.com",
  "your-password"
);

// Retrieve the ID token
const idToken = await userCredential.user.getIdToken();
```

```typescript TypeScript
import { getAuth, signInWithEmailAndPassword } from "firebase/auth";

const auth = getAuth();
const userCredential = await signInWithEmailAndPassword(
  auth,
  "user@example.com",
  "your-password"
);

const idToken: string = await userCredential.user.getIdToken();
```

</CodeGroup>

## Passing the token to the API

Include the token in every request as a `Bearer` token in the `Authorization` header:

```
Authorization: Bearer <firebase_id_token>
```

<CodeGroup>

```bash cURL
curl --request GET \
  --url https://api.paypac.com.co/api/auth/me \
  --header "Authorization: Bearer <firebase_id_token>"
```

```javascript JavaScript (fetch)
const response = await fetch("https://api.paypac.com.co/api/auth/me", {
  method: "GET",
  headers: {
    Authorization: `Bearer ${idToken}`,
  },
});

const profile = await response.json();
```

</CodeGroup>

## Token expiration and refresh

Firebase ID tokens expire after **1 hour**. To avoid `401` errors in long-running sessions, refresh the token before it expires:

<Steps>
  <Step title="Force-refresh the token">
    Pass `true` to `getIdToken()` to force Firebase to issue a new token, even if the current one has not yet expired:

    ```javascript JavaScript
    // Force a token refresh
    const freshToken = await user.getIdToken(true);
    ```
  </Step>

  <Step title="Use onIdTokenChanged for automatic refresh">
    Listen for token changes so your app always has a valid token without polling:

    ```javascript JavaScript
    import { getAuth, onIdTokenChanged } from "firebase/auth";

    const auth = getAuth();

    onIdTokenChanged(auth, async (user) => {
      if (user) {
        const idToken = await user.getIdToken();
        // Store idToken and use it for API requests
      }
    });
    ```
  </Step>
</Steps>

<Warning>
  If you send an expired token, the backend returns `401 Unauthorized` with `{ "message": "Token expirado" }`. Refresh the token and retry the request.
</Warning>

## How the middleware verifies requests

Every protected route runs through the `authenticate` middleware. Here is exactly what it does:

<Steps>
  <Step title="Extract the token from the Authorization header">
    The middleware checks for a header in the form `Bearer <token>`. If the header is missing or malformed, it returns `401` immediately.

    ```typescript src/middlewares/auth.middleware.ts
    const authHeader = req.headers.authorization;

    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      res.status(401).json({ message: 'Token no proporcionado' });
      return;
    }

    const idToken = authHeader.split(' ')[1];
    ```
  </Step>

  <Step title="Verify the token with Firebase Admin SDK">
    The token is verified by calling `firebaseAuth.verifyIdToken()`. The Firebase Admin SDK caches Google's public keys (JWKS) after the first call, so subsequent verifications run as fast local cryptographic operations (~2–5 ms).

    ```typescript src/middlewares/auth.middleware.ts
    const decodedToken = await firebaseAuth.verifyIdToken(idToken);
    const firebaseUid = decodedToken.uid;
    ```
  </Step>

  <Step title="Look up the user in the database">
    The middleware resolves the Firebase UID to a PayPac user record using a unique-index lookup (`findUnique`), which is an O(1) operation:

    ```typescript src/middlewares/auth.middleware.ts
    const user = await prisma.user.findUnique({
      where: { firebase_uid: firebaseUid },
    });

    if (!user) {
      res.status(401).json({ message: 'Usuario no encontrado en el sistema' });
      return;
    }
    ```
  </Step>

  <Step title="Attach the user to the request">
    The resolved user record is attached to `req.user` and is available to all subsequent middleware and route handlers.

    ```typescript src/middlewares/auth.middleware.ts
    req.user = user;
    next();
    ```
  </Step>
</Steps>

## Role authorization

After authentication, some endpoints require specific roles. Role checking is handled by the `authorizeRoles` middleware, which runs after `authenticate`:

```typescript src/middlewares/role.middleware.ts
export const authorizeRoles = (...allowedRoles: string[]) => {
  return (req: Request, res: Response, next: NextFunction): void => {
    const user = req.user;

    if (!user || !allowedRoles.includes(user.role)) {
      res.status(403).json({ message: 'Acceso denegado: rol no autorizado' });
      return;
    }

    next();
  };
};
```

PayPac defines six roles:

| Role             | Description                                              |
| ---------------- | -------------------------------------------------------- |
| `PAYPAC`         | Platform superadmin. Full access to all resources.       |
| `ORGANIZER`      | Event organizer. Manages events and staff for a company. |
| `STAFF`          | Event staff member. Operational access within an event.  |
| `STAFF_PROMOTER` | Staff member with promoter capabilities.                 |
| `PROMOTER`       | Ticket promoter. Sells tickets for assigned events.      |
| `CUSTOMER`       | End user. Purchases tickets.                             |

<Tip>
  Roles are stored server-side in the PayPac database. You cannot escalate your own role by modifying a token — the backend always reads the role from the database record, not from the token payload.
</Tip>

For example, `POST /api/auth/new-user` requires the `PAYPAC` or `ORGANIZER` role:

```typescript src/routes/auth.routes.ts
router.post(
  '/new-user',
  authenticate,
  authorizeRoles('PAYPAC', 'ORGANIZER'),
  validateRequest(registerUserSchema),
  register
);
```

## Error responses

| Status | Message                            | Cause                                               |
| ------ | ---------------------------------- | --------------------------------------------------- |
| `401`  | `Token no proporcionado`           | `Authorization` header is missing or not `Bearer`.  |
| `401`  | `Token expirado`                   | The Firebase ID token has expired (>1 hour old).    |
| `401`  | `Token malformado`                 | The token string is not a valid JWT.                |
| `401`  | `Token inválido`                   | Token verification failed for any other reason.     |
| `401`  | `Usuario no encontrado en el sistema` | Token is valid but no matching user exists in the DB. |
| `403`  | `Acceso denegado: rol no autorizado` | User is authenticated but lacks the required role. |
